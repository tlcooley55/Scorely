User Story 1 - As a user, I want to create an account so that I can store my music ratings.

User Story 2 - As a user, I want to search for songs so that I can find music to rate.

User Story 3 - As a user, I want to rate songs from 1–5 stars so that I can record my opinion.

User Story 4 - As a user, I want to write reviews about songs so that I can share my thoughts.

User Story 5 - As a user, I want to filter my listening history by songs I rated 4 or 5 stars so that I can revisit my top picks.

User Story 6 - As a user, I want to view my listening history so that I can see songs I rated before.

User Story 7 - As a user, I want to view a friend’s profile so that I can see their ratings.

User Story 8 - As a user, I want to see a friend’s top five songs so that I can discover music.

Rejected User Stories

User Story 9 - As a user, I want to log into my account so that I can access my music history. Rejected – This story describes a basic authentication requirement rather than a user-facing feature. Logging into an account is necessary for the system to function but does not represent a meaningful interaction related to the core purpose of the app (rating and tracking music).

User Story 10 - As a user, I want to delete ratings so that I can remove songs from my profile. Rejected – This functionality overlaps heavily with editing ratings. If users can edit ratings, they could simply change the rating instead of deleting it. Including both features would introduce unnecessary redundancy.

User Story 11 - As a user, I want to sort my listening history by date so that I can see recent ratings. Rejected – Sorting methods are more related to interface design decisions than core system functionality. Sorting behavior can be handled during interface design rather than treated as a primary user story.

User Story 12 - As a user, I want to sort songs by genre so that I can organize my music catalog. Rejected – While useful, this feature was not emphasized by the client and would require additional song metadata. To keep the first version simple, it was considered lower priority.

User Story 13 - As a user, I want to bookmark songs so that I can revisit them later. Rejected – Bookmarking songs is not essential to the main goal of rating and tracking music. The client focused more on rating songs and viewing history.

User Story 14 - As a user, I want to view song artwork so that I can visually recognize songs. Rejected – This story focuses on visual presentation rather than functionality. Artwork would be handled later during interface design.

User Story 15 - As a user, I want to remove friends so that I can control who I interact with. Rejected – The client did not emphasize managing friend relationships beyond adding friends and viewing profiles. To keep the system minimal, this feature was excluded.

User Story 16 - As a user, I want to receive notifications when friends rate songs. Rejected – The client specifically stated that notifications were not necessary for the first version of the system.

User Story 17 - As a user, I want to compare my ratings with friends so that I can see shared taste. Rejected – This feature would require additional logic to analyze rating patterns between users and was considered too complex for the first version.

User Story 18 - As a user, I want to rate albums so that I can evaluate full projects. Rejected – The client suggested simplifying the system by focusing mainly on songs rather than songs, albums, and artists.

User Story 19 - As a user, I want to rate artists so that I can track my favorite musicians. Rejected – Similar to album ratings, artist ratings were removed to keep the scope manageable.

User Story 20 - As a user, I want trending songs to appear so that I can discover popular music. Rejected – Implementing trending songs would require algorithms and broader system data, which are outside the scope of this project.

User Story 21 - As a user, I want to share playlists so that friends can listen to music I recommend. Rejected – The application is focused on rating and tracking music rather than acting as a music sharing or streaming platform.

User Story 22 - As a user, I want to link my Spotify account so that ratings update automatically. Rejected – External integrations such as Spotify would likely be technically difficult and outside the scope of the project.

User Story 23 - As a user, I want the system to recommend songs so that I can discover new music. Rejected – Recommendation systems require algorithms and data analysis that are beyond the scope of the project. The client emphasized friend-based discovery instead.

User Story 24 - As a user, I want to organize my rated songs into custom lists so that I can group music by mood or activity. Rejected – Custom list organization adds additional interface complexity and was not part of the client’s core requirements for the first version.

User Story 25 - As a user, I want to edit my written reviews so that I can update my opinions later. Rejected – While useful, review editing was considered secondary functionality and was excluded to keep the initial system simpler.

User Story 26 - As a user, I want to see statistics about how many songs I rated each month so that I can track my listening habits. Rejected – Analytical statistics require additional data processing and were not emphasized by the client.

User Story 27 - As a user, I want to export my ratings to a file so that I can keep a personal backup of my music preferences. Rejected – Data export is not necessary for the primary goal of the application and would increase development complexity.

User Story 28 - As a user, I want to customize my profile appearance so that my account reflects my personality. Rejected – Profile customization focuses on visual design rather than the main functionality of rating and discovering music.

User Story 29 - As a user, I want to see the most reviewed songs among my friends so that I can quickly discover popular music within my social circle. Rejected – This feature requires additional aggregation and ranking logic, which was considered too complex for the initial version of the system.
